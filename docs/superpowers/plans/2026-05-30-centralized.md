# План реализации: Централизованный модуль (под-проект №6)

**Статус:** черновик · **Дата:** 2026-05-30

## Обзор

Поэтапная реализация по спецификации `2026-05-30-centralized.md`. Клиент
проектируется под **целевой контракт «токен + WebSocket»**; серверную реализацию
ведёт агент в `vsc_web` по инструкции (этап 7). Каждый этап заканчивается
зелёными тестами и осмысленным коммитом (на русском, без AI-авторства).

Тестируем против in-process фейкового сервера (`httpx.MockTransport` для REST +
`websockets.serve` для WS) — не зависим от готовности `vsc_web`.

## Этапы

### Этап 1 — Каркас пакета, модели, ошибки

- `src/mys_centralized/{__init__,models,errors}.py`.
- Dataclasses: `Session`, `Room`, `RemoteMessage`, `SyncCursor`.
- Иерархия ошибок: `AuthError`, `NetworkError`, `ServerError`, `ProtocolError`.
- Тесты: конструкция моделей, сериализация курсора.
- **Коммит:** «feat: каркас централизованного модуля — модели и ошибки (№6, ч.1)».

### Этап 2 — REST-клиент

- `api_client.py`: `RestClient(base_url, *, client=None)` поверх
  `httpx.AsyncClient`; методы `register/login/logout/list_rooms/get_messages/
  post_message`; bearer-заголовок; маппинг статусов в ошибки (§5).
- Тесты (`httpx.MockTransport`): успех/ошибки каждого метода, пагинация
  `next_cursor`, идемпотентность `post_message`, 401/4xx/5xx/битый JSON.
- **Коммит:** «feat: REST-клиент централизованного режима (№6, ч.2)».

### Этап 3 — WebSocket-клиент

- `ws_client.py`: `WsClient(url, token)`; connect → auth-кадр → `ready`;
  async-итератор событий; реконнект с экспоненциальным backoff + джиттер;
  `ping_interval`.
- Тесты (`websockets.serve` на loopback): рукопожатие (ready/ошибка), приём
  `message`, реконнект после обрыва, авторизация неверным токеном.
- **Коммит:** «feat: WebSocket-клиент централизованного режима (№6, ч.3)».

### Этап 4 — Персист и движок синхронизации

- При необходимости миграция схемы vault (например, поля для `client_msg_id`/
  состояния) — поднять `SCHEMA_VERSION` и добавить шаг в `migrations.py`; иначе
  обойтись детерминированными `msg_id` и `settings`-курсорами (см. спеку §4).
- `sync.py`: `SyncEngine` — первичная загрузка комнат+истории по курсору, переход
  на live, дедуп (серверный `id` + `client_msg_id`), персист в vault, продвижение
  курсоров, колбэки.
- Хранение сессии аккаунта в `settings`; токен в памяти — `SecureBytes`.
- Тесты: дедуп, продвижение курсора, дозагрузка «после оффлайна», персист во
  временный vault (как в интеграционных тестах хранилища).
- **Коммит:** «feat: синхронизация и персист централизованного режима (№6, ч.4)».

### Этап 5 — Оркестратор-сервис

- `service.py`: `CentralizedService` — asyncio loop в фоновом потоке (по образцу
  `P2PService`); `login/start/stop/send_message`; потокобезопасные вызовы и
  колбэки в UI-поток; общий доступ к vault через `_LockedConnection`.
- Тесты-интеграция: in-process фейковый сервер → `login` → синк → live-приём →
  отправка (идемпотентная) → проверка vault; реконнект закрывает пропуски.
  Блокирующие методы гонять через `asyncio.to_thread`.
- **Коммит:** «feat: оркестратор централизованного сервиса (№6, ч.5)».

### Этап 6 — Интеграция в контроллер и UI

- `AppController`: фабрика/жизненный цикл `CentralizedService`, `central_login`,
  `central_send`, переключение `AppMode.CENTRALIZED`.
- UI «Центр»: форма входа (server_url=`https://soufos.ru`, username, password,
  Вход/Регистрация) при отсутствии сессии; иначе список комнат + чат; ошибки
  входа на экран; real-time через сигналы Qt.
- Тесты headless (`QT_QPA_PLATFORM=offscreen`): вход с фейковым сервисом,
  отображение комнат/сообщений, отправка.
- **Коммит:** «feat: интеграция централизованного режима в контроллер и UI (№6, ч.6)».

### Этап 7 — Инструкция для сервера `vsc_web`

- `docs/superpowers/specs/2026-05-30-server-centralized-api.md` — самодостаточная
  инструкция агенту в `vsc_web` (по образцу `2026-05-30-server-rendezvous-relay.md`):
  точный контракт §5–§6, токен-таблица, идемпотентность, эталонные эндпоинты
  (flask-sock для `/ws`), Bearer-вариант существующих REST + `room_id`/пагинация,
  деплой за nginx, тесты. Form/cookie-маршруты веб-версии не ломать.
- Кодеки/формы тел сверить байт-в-байт с клиентом.
- **Коммит:** «docs: инструкция по серверному токен+WebSocket API для vsc_web (№6, ч.7)».

- Прогнать весь набор тестов; обновить `CLAUDE.md` (отметить готовность №6,

## Риски и решения

- **Сервер ещё не реализует контракт.** Клиент полностью тестируется фейковым
  сервером; против реального `vsc_web` — после этапа 7 (в его репозитории).
- **Расхождение контракта клиент↔сервер.** Контракт зафиксирован в спеке §5–§6 и
  продублирован в серверной инструкции; сверка тел байт-в-байт (как в №5).
- **Порядок отправки (как у ratchet в №4) здесь неактуален** — централизованный
  режим без ratchet, любой участник шлёт в любой момент.
- **Миграция схемы.** Если потребуются новые колонки — строго через
  `migrations.py` с бампом `SCHEMA_VERSION`, чтобы существующие vault мигрировались.

## Критерии готовности

См. спецификацию §13. Кратко: пакет `mys_centralized` (REST+WS+sync+service),
персист сессии/истории в зашифрованном vault (токен не на диске вне vault),
зелёный интеграционный сценарий логин→синк→live→отправка с реконнектом,
соблюдение модульных границ, инструкция для `vsc_web`.
