# Спецификация: Централизованный модуль (под-проект №6)

**Статус:** черновик · **Дата:** 2026-05-30 · **Зависит от:** хранилище (№2),
UI-каркас (№3) · **Сервер:** доработки в `vsc_web` (см. §11)

## 1. Цель и границы

Дать пользователю работу с аккаунтом на сервере `soufos.ru` (веб-версия —
`vsc_web`): вход/регистрация, список комнат, история и real-time обмен
сообщениями, с локальным кэшем в зашифрованном vault (история доступна офлайн на
чтение). Это **не** анонимный E2E-режим: в централизованном режиме сервер хранит
историю и видит открытый текст — by design (анонимность только на уровне
регистрации без личных данных). E2E-секреты централизованного режима в v1 не
вводятся.

### В объёме

- **Целевой контракт сервера: токен + WebSocket** (по дизайн-доку), а не текущие
  cookie-сессии `vsc_web`. Серверная реализация контракта выносится в `vsc_web`
  отдельной инструкцией (как было с под-проектом №5); клиент проектируется под
  этот контракт.
- REST-клиент (`httpx`): регистрация, логин/логаут (bearer-токен), список комнат,
  история комнаты (с курсором), отправка сообщения (идемпотентно по
  `client_msg_id`).
- WebSocket-клиент (`websockets`): подписка с токен-авторизацией, приём real-time
  событий, авто-реконнект с backoff.
- Движок синхронизации: первичная загрузка (комнаты + история по курсору) →
  переход на live через WS; дедуп по `client_msg_id`/серверному `id`; персист в
  vault (`conversations`/`messages`); хранение курсоров.
- Оркестратор `service.py` (asyncio в фоновом потоке, по образцу `P2PService`) и
  интеграция в контроллер/UI («Центр»: форма входа → список+чат).
- Хранение сессии аккаунта (server_url, username, user_id, token) в зашифрованном
  vault.
- Тесты: юнит (REST через `httpx.MockTransport`, WS через локальный
  `websockets.serve`), интеграционные (логин → синк → приём → отправка через
  in-process фейковый сервер), негативные (неверный пароль, разрыв WS, 401/5xx).

### Вне объёма (v1)

- E2E-шифрование централизованных сообщений (сервер видит текст — by design).
- Создание/управление комнатами и приглашения (только потребление существующих
  комнат; создание — follow-up).
- Передача файлов, реакции, прочтения, редактирование/удаление сообщений.
- Пуш-уведомления, оффлайн-отправка с очередью переживающей перезапуск (минимум —
  пометка `pending`/`failed`, без гарантий доставки).

## 2. Контекст и зависимости

Модуль не знает о Qt; экспонирует API для `mys_ui`. Зависит от:

- `mys_storage`: персист сессии аккаунта, комнат (`conversations`), сообщений
  (`messages`), настроек/курсоров (`settings`).
- `httpx` (REST) и `websockets` (WS) — уже в зависимостях проекта.
- TLS обязателен: только `https://`/`wss://` с проверкой сертификатов.

Границы модулей (CLAUDE.md): сеть не знает деталей хранилища сверх явного
репозиторного API; крипто здесь не участвует (нет E2E в централизованном v1).

Существующий сервер `vsc_web` сейчас использует cookie-сессии Flask, JSON-логина и
WebSocket не имеет. Поэтому контракт §5–§7 — **целевой**, его реализует агент в
`vsc_web` по инструкции из §11. До готовности сервера клиент тестируется против
in-process фейкового сервера (§10).

## 3. Компоненты и файлы

Новый пакет `src/mys_centralized/`:

- `models.py` — dataclasses: `Account`/`Session` (server_url, username, user_id,
  token), `Room`, `RemoteMessage`, `SyncCursor`.
- `errors.py` — типизированные ошибки: `AuthError`, `NetworkError`,
  `ServerError`, `ProtocolError`.
- `api_client.py` — `RestClient` поверх `httpx.AsyncClient`: `register`, `login`,
  `logout`, `list_rooms`, `get_messages(room_id, after=…, limit=…)`,
  `post_message(room_id, body, client_msg_id)`. Маппинг HTTP-статусов в ошибки.
- `ws_client.py` — `WsClient`: connect+auth, async-итератор событий, reconnect с
  экспоненциальным backoff и джиттером.
- `sync.py` — `SyncEngine`: первичная загрузка, переход на live, дедуп, персист в
  vault, ведение курсоров; колбэки на новое сообщение / смену статуса соединения.
- `service.py` — `CentralizedService`: event loop в фоновом потоке; методы для
  контроллера (`login`, `start`, `stop`, `send_message`) и колбэки в UI-поток.
- `__init__.py` — публичный экспорт.

Тесты: `tests/centralized/` (юнит + интеграция, фейковый сервер в `conftest`).

## 4. Модель данных и персист

Переиспользуем существующую схему vault (`SCHEMA_VERSION` поднять при
необходимости — см. план):

- **Комната → conversation.** `conv_id = f"central:{room_id}"`,
  `mode="centralized"`, `title = room.name`. Через `upsert_conversation`.
- **Сообщение → message.** Детерминированный `msg_id = f"central:{room_id}:{server_id}"`
  для входящих/подтверждённых; для исходящих до подтверждения —
  `msg_id = f"central:{room_id}:c:{client_msg_id}"`, после ответа сервера запись
  обновляется (или переносится) на детерминированный серверный id. `sender` —
  username, `body` — открытый текст, `state ∈ {pending, sent, failed}`.
- **Сессия аккаунта.** В `settings`: `central.server_url`, `central.username`,
  `central.user_id`, `central.token`. Токен чувствителен, но vault зашифрован;
  в памяти держать в `SecureBytes` (как ключ БД). Никогда не писать токен на диск
  вне зашифрованного vault.
- **Курсоры синка.** В `settings`: `central.cursor.<room_id>` — серверный id/ts
  последнего загруженного сообщения; используется для дозагрузки истории и
  отбрасывания дублей из WS.

`client_msg_id` — случайный UUID, генерируется клиентом при отправке; даёт
идемпотентность на сервере и дедуп эхо из WS.

## 5. REST-контракт (целевой)

База: `https://soufos.ru`. Авторизация — `Authorization: Bearer <token>` (кроме
register/login). Тела — JSON, время — ISO-8601 UTC.

- `POST /api/auth/register` `{username, password}` → `201 {token, user:{id, username}}`
  · конфликт имени → `409`.
- `POST /api/auth/login` `{username, password}` → `200 {token, user:{id, username}}`
  · неверные данные → `401`.
- `POST /api/auth/logout` (Bearer) → `204`.
- `GET /api/rooms` (Bearer) → `200 {rooms:[{id, name, is_direct, updated_at}]}`.
- `GET /api/rooms/{id}/messages?after={cursor}&limit={n}` (Bearer) →
  `200 {messages:[{id, room_id, sender, body, created_at}], next_cursor}`
  · `after` опционально (без него — с начала/последние `limit`); `next_cursor=null`
  когда больше нет.
- `POST /api/messages` (Bearer) `{room_id, body, client_msg_id}` →
  `200 {id, room_id, sender, body, created_at}` · повтор с тем же `client_msg_id`
  возвращает ту же запись (идемпотентность), не создаёт дубль.

Имя поля времени `created_at` (ISO-8601) выбрано для совпадения с уже
существующими JSON-эндпоинтами `vsc_web` — минимизируем расхождение. Текущие
эндпоинты возвращают `{id, sender, body, created_at}` без `room_id`; целевой
контракт добавляет `room_id` (для маршрутизации в conversation) и пагинацию.

Ошибки единым телом: `{error: "<code>"}`. Маппинг: `401→AuthError`,
`4xx→ServerError`, `5xx/таймаут/обрыв→NetworkError`, неожиданная форма →
`ProtocolError`.

## 6. WebSocket-контракт (целевой)

Эндпоинт `wss://soufos.ru/ws` (отдельно от P2P-relay `wss://soufos.ru/p2p` из
№5). Кадры — JSON-объекты с полем `type`.

1. Клиент подключается и шлёт первый кадр: `{"type":"auth","token":"<token>"}`.
2. Сервер отвечает `{"type":"ready"}` (успех) или `{"type":"error","code":"unauthorized"}`
   и закрывает соединение.
3. После `ready` сервер пушит новые сообщения по всем комнатам пользователя:
   `{"type":"message","room_id":…,"id":…,"sender":…,"body":…,"created_at":…}`.
4. Keepalive — стандартный WS ping/pong (`websockets` сам отвечает на ping;
   клиент включает `ping_interval`).

Отправка сообщений в v1 идёт **через REST** `POST /api/messages` (чистое
разделение: REST пишет, WS читает). Эхо собственного сообщения приходит и через
WS — дедуп по `client_msg_id`/серверному `id`.

## 7. Поток работы

1. **Вход.** UI отдаёт server_url, username, password → `service.login(...)`:
   REST `login` (или `register`) → токен; сессия сохраняется в vault.
2. **Старт синка.** `SyncEngine`: `GET /api/rooms` → upsert комнат; по каждой
   комнате дозагрузка истории от `cursor` через `get_messages` (страницами по
   `next_cursor`) → персист, продвижение курсора.
3. **Live.** `WsClient` подключается, авторизуется; входящие `message` → дедуп по
   курсору/`client_msg_id` → персист → колбэк в UI; курсор продвигается.
4. **Отправка.** `service.send_message(conv_id, body)`: запись `pending` локально с
   `client_msg_id` → REST `post_message` → по ответу запись переводится в `sent`
   с серверным id; при ошибке — `failed`. Эхо из WS дедупится.
5. **Реконнект.** При обрыве WS — backoff-переподключение; после `ready` —
   повторная дозагрузка истории от курсора (закрывает пропуски за время оффлайна),
   затем снова live.
6. **Логаут.** REST `logout` (best-effort) → очистка сессии в vault; локальный кэш
   истории остаётся (доступен офлайн на чтение) либо чистится по настройке
   (follow-up).

## 8. Параллелизм и потоки

По образцу `P2PService`: `CentralizedService` поднимает свой asyncio event loop в
фоновом потоке. Публичные методы для контроллера потокобезопасны
(`run_coroutine_threadsafe`), колбэки в UI диспетчеризуются как у P2P. Доступ к
vault — через общий `_LockedConnection` (vault открыт с
`check_same_thread=False`), так что UI-поток (чтение) и поток сервиса (запись)
делят один Vault (см. CLAUDE.md «Доступ к vault из потоков»).

## 9. Интеграция с контроллером/UI

- `AppController`: фабрика и жизненный цикл `CentralizedService` (по образцу
  `_p2p_factory`); методы `central_login`, `central_send`, переключение
  `AppMode.CENTRALIZED`.
- UI «Центр»: если сессии нет — форма входа (server_url по умолчанию
  `https://soufos.ru`, username, password, кнопки Вход/Регистрация); если есть —
  список комнат + чат, как у P2P-панели. Ошибки входа показываются пользователю
  (неверный пароль / сеть).
- Real-time: колбэк сервиса обновляет список/чат через сигналы Qt (как P2P).

## 10. Тестирование

- **Юнит REST** (`httpx.MockTransport`): успех и ошибки login/register/logout,
  list_rooms, get_messages (пагинация по `next_cursor`), post_message
  (идемпотентность), маппинг статусов в ошибки.
- **Юнит WS** (`websockets.serve` на loopback): auth-рукопожатие (ready/ошибка),
  приём `message`, реконнект с backoff после обрыва.
- **Sync**: дедуп (повтор id и эхо `client_msg_id`), продвижение курсора,
  дозагрузка после «оффлайна», персист во временный vault.
- **Интеграция**: in-process фейковый сервер (REST через `MockTransport` + WS
  через `websockets.serve`) → `service.login` → синк → приём live → отправка →
  проверка состояния vault. Блокирующие вызовы сервиса в тестах гонять через
  `asyncio.to_thread`, чтобы не стопорить loop теста (как в №4).
- **Негативные**: 401 на старый токен, 5xx/таймаут, битый JSON, обрыв WS на
  середине, отправка при недоступном сервере (`pending→failed`).
- `pytest-asyncio` (`asyncio_mode=auto` в `pytest.ini`) уже настроен.

## 11. Серверные доработки (`vsc_web`)

**Текущее состояние `vsc_web` (сверено по исходникам, не по предположениям):**
аутентификация — cookie-сессии Flask (`session["user_id"]`), form-эндпоинты
`/register`, `/login`, `/logout` (без JSON, без токена). REST уже есть, но под
cookie-сессией: `GET /api/rooms` → `{rooms:[{id,name,is_direct}]}`,
`GET /api/rooms/<id>/messages` → `{messages:[{id,sender,body,created_at}]}`
(без `room_id`, без пагинации), `POST /api/messages` `{room_id, body}` →
`{id,sender,body,created_at}` (без идемпотентности). WebSocket отсутствует.
Модели (`User.check_password`, `Room.messages`/`Message.room` back_populates)
корректны. Зависимости — `flask`, `Flask-SQLAlchemy` (SQLite). `rooms.py` —
заглушка (только namespace blueprint).

**Что добавить для целевого контракта** (рядом с существующим, не ломая
веб-версию):

Реализацию ведёт агент в `vsc_web` по самодостаточной инструкции (как
`2026-05-30-server-rendezvous-relay.md` для №5). Инструкция — отдельный
deliverable фазы реализации; здесь фиксируем требования:

- Токен-таблица (`tokens`/`sessions`): случайный opaque-токен на вход, проверка по
  `Authorization: Bearer`; логаут инвалидирует токен.
- JSON-эндпоинты `/api/auth/*`, `/api/rooms*`, `/api/messages` (рядом с текущими
  form/cookie-маршрутами; их не ломать — веб-версия продолжает работать).
- Идемпотентность `POST /api/messages` по `client_msg_id` (колонка + uniq на
  `(room_id, client_msg_id)`).
- WebSocket `/ws` (flask-sock, как `/p2p` в №5): auth-кадр → `ready` → пуш
  `message` подписчикам комнаты при создании сообщения (включая созданные через
  REST).
- Bearer-вариант существующих REST-эндпоинтов: к `/api/rooms`,
  `/api/rooms/<id>/messages`, `/api/messages` добавить авторизацию по токену
  (помимо cookie), `room_id` и пагинацию (`after`/`limit`/`next_cursor`) в
  историю. Form/cookie-маршруты веб-версии не трогать.
- Зависимости: добавить `flask-sock` (WS), при желании `Flask-Migrate` для
  миграции схемы под токен-таблицу и `client_msg_id`.
- Деплой за nginx с проксированием WS (как `/p2p`); тесты контракта.

## 12. Follow-up и техдолг

- Создание/управление комнатами и приглашения из десктопа.
- Надёжная очередь исходящих (переживающая перезапуск) и ретраи.
- Прочтения/доставки, реакции, редактирование/удаление.
- Опциональный E2E поверх централизованного режима (за пределами v1; сервер тогда
  хранит шифротекст — конфликтует с «история на сервере», требует отдельного
  дизайна).
- Очистка локального кэша при логауте по настройке.

## 13. Критерии готовности

- Пакет `src/mys_centralized/` с REST+WS клиентами, sync-движком и сервисом.
- Сессия аккаунта и кэш истории персистятся в зашифрованном vault; токен не
  попадает на диск вне vault.
- Интеграционный тест: логин → синк комнат/истории → приём live по WS → отправка
  (идемпотентная) → корректное состояние vault; реконнект закрывает пропуски.
- Все тесты зелёные; модульные границы соблюдены (нет утечки деталей сети/хранилища).
- Инструкция для `vsc_web` с точным контрактом §5–§6 (фаза реализации).
