# UI-каркас (mys_ui) — спецификация

**Дата:** 2026-05-29
**Статус:** черновик (фаза дизайна под-проекта №3)
**Контекст:** под-проект №3 из `docs/superpowers/specs/2026-05-29-mys-desktop-design.md`
(раздел 3 — UI-слой и контроллер). Зависит от хранилища `mys_storage` (под-проект
№2, готов). Сетевые модули (№4 децентрализованный, №6 централизованный) ещё не
реализованы.

## 1. Цель и границы

Функциональный **каркас** настольного приложения на PySide6: окна, навигация,
разблокировка/создание зашифрованного vault, двухпанельный главный экран
(список диалогов + чат), переключение режимов. Визуально — **голый скелет** без
оформления (пользователь сделает дизайн позже); важна **полная
функциональность обвязки**, а не внешний вид.

**Что реально работает (через `mys_storage`):**
- создание vault и разблокировка по мастер-паролю (Argon2id, лимит попыток,
  блокировка);
- список диалогов и сообщений читаются из БД;
- отправка сообщения сохраняется в БД (локально);
- переключение режима, блокировка (закрытие vault → возврат к экрану входа).

**Что заглушка (модулей ещё нет):**
- сетевая передача сообщений (P2P/сервер) — №4/№6;
- PAKE-хендшейк по секретной фразе, Double Ratchet поверх транспорта — экран
  ввода фразы существует, но создаёт диалог локально без реального крипто-канала.

**Границы (CLAUDE.md):** тонкий UI поверх контроллера. UI **не знает** ни крипто-,
ни сетевых деталей, ни SQL. Весь доступ к данным — через **контроллер приложения**,
который инкапсулирует `mys_storage`. Контроллер не импортирует Qt.

## 2. Стек и зависимости

- **GUI:** `PySide6>=6.11` (системный `qt6-base 6.11` присутствует).
- **Тесты UI:** `pytest-qt`, запуск headless через `QT_QPA_PLATFORM=offscreen`.
- **Данные:** `mys_storage` (готов).

Добавляется пакет `src/mys_ui/`. В `pyproject.toml`: `PySide6` в основные
зависимости, `pytest-qt` — в `dev`.

> Замечание про async: сетевых модулей ещё нет, поэтому каркас **синхронный**
> (контроллер обращается к хранилищу напрямую). Интеграция `asyncio`
> (напр. `qasync`) с Qt-циклом — задача под-проекта №4, здесь только закладывается
> точка расширения в контроллере.

## 3. Архитектура

```
┌──────────────────────────────────────────────┐
│ UI (PySide6): UnlockWindow, MainWindow,        │
│ ConversationList, ChatView, MessageInput,      │
│ TopBar, SettingsDialog, PhraseDialog           │
└───────────────┬────────────────────────────────┘
                │ сигналы/слоты, вызовы методов
┌───────────────▼────────────────────────────────┐
│ AppController (без Qt): держит Vault, режим,     │
│ текущий диалог; методы создания/разблокировки,   │
│ списки диалогов/сообщений, отправка               │
└───────────────┬────────────────────────────────┘
                │
        ┌───────▼────────┐
        │   mys_storage   │ (Vault, репозитории)
        └─────────────────┘
```

- **AppController** — чистый Python, тестируется без Qt. Единая точка состояния
  сессии: `vault`, `mode`, `current_conversation`.
- **UI-виджеты** — отображение и ввод; дергают контроллер, показывают результат.
  Никакого прямого `mys_storage`/SQL в виджетах.

## 4. Экраны и потоки

### 4.1 Запуск → вход
1. Определить путь vault (см. §8, `default_vault_path()`).
2. Если vault **не существует** → экран **создания** (поле пароля + подтверждение).
3. Если **существует** → экран **разблокировки** (поле пароля).
4. Неверный пароль → сообщение об ошибке; при срабатывании лимита — показать, что
   вход заблокирован на N секунд (`VaultLocked.seconds_left`).
5. Успех → главное окно.

### 4.2 Главное окно (две панели + верхняя панель)
- **Верхняя панель:** переключатель режима `[Центр | P2P]`, кнопка «Настройки»,
  кнопка «Блокировка 🔒».
- **Левая панель:** список диалогов текущего режима (`+ Новый диалог`).
- **Правая панель:** заголовок диалога, лента сообщений, поле ввода + «Отправить».

### 4.3 Действия
- **Выбор диалога** → лента сообщений из БД.
- **Отправка** → сообщение сохраняется в БД как исходящее (`direction="out"`,
  `status="local"` — сеть не подключена), лента обновляется.
- **+ Новый диалог** → в P2P открывается экран ввода секретной фразы (стаб: создаёт
  диалог локально, без PAKE); в централизованном — создаёт диалог-заглушку.
- **Переключение режима** → список диалогов фильтруется по `mode`.
- **Блокировка** → `controller.lock()` (закрытие vault, зануление ключа) → возврат
  к экрану входа.
- **Настройки** (минимум): сменить мастер-пароль; переключатель темы; флаг
  duress-wipe. Без избыточности.

## 5. Контроллер — интерфейс (эскиз)

```python
class AppController:
    vault: Vault | None
    mode: str               # "decentralized" | "centralized"

    def vault_exists(self) -> bool
    def create_vault(self, password: bytes) -> None
    def unlock(self, password: bytes) -> None     # raises WrongPassword, VaultLocked
    def lock(self) -> None                         # close vault, wipe key
    def set_mode(self, mode: str) -> None
    def list_conversations(self) -> list[dict]     # текущего режима
    def create_conversation(self, title: str, *, room_phrase: str | None = None) -> int
    def list_messages(self, conversation_id: int) -> list[dict]
    def send_message(self, conversation_id: int, text: str) -> int   # сохраняет локально
    def change_password(self, old: bytes, new: bytes) -> None
```

Тело (`text`) кодируется в `bytes` для хранения; обратно — для показа. Контроллер
**не знает** про Qt.

## 6. Тема оформления

Минимальная **тёмная** тема по умолчанию: применить `Fusion` + тёмную `QPalette`
на уровне `QApplication`. Без кастомного QSS и сложных стилей — «чтобы глаза не
горели». Переключатель темы (тёмная/светлая) — в настройках, значение хранится в
`vault.settings` (ключ `theme`).

## 7. Тестирование

Headless через `QT_QPA_PLATFORM=offscreen` (выставляется в `conftest.py`).

- **Контроллер (без Qt):** создание/разблокировка против временного vault; неверный
  пароль → `WrongPassword`; лимит → `VaultLocked`; `lock()` закрывает vault;
  список диалогов фильтруется по режиму; `send_message` сохраняет и читается назад;
  смена пароля.
- **UI (pytest-qt):** на экране входа неверный пароль показывает ошибку, верный —
  открывает главное окно; список диалогов наполняется из БД; отправка добавляет
  сообщение в ленту; переключение режима меняет список; «Блокировка» возвращает к
  входу.

## 9. Открытые вопросы / отложено

- Реальная сетевая отправка и крипто-канал (ratchet/transport) — под-проект №4;
  до тех пор `send_message` только сохраняет локально, `status="local"`.
- PAKE по секретной фразе (`PhraseDialog` сейчас стаб) — под-проект №4.
- `asyncio`/`qasync`-интеграция с Qt-циклом — под-проект №4.
- Полноценный дизайн/оформление — делает пользователь поверх готового каркаса.
- Централизованный вход (логин/токен на сервере) — под-проект №6; в каркасе режим
  «Центр» оперирует только локальными диалогами-заглушками.
