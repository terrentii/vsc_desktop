# Локальное хранилище (mys_storage) — спецификация

**Дата:** 2026-05-29
**Статус:** черновик (фаза дизайна под-проекта №2)
**Контекст:** под-проект №2 из дизайн-документа
`docs/superpowers/specs/2026-05-29-mys-desktop-design.md` (раздел 6). Зависит от
крипто-ядра `mys_crypto` (под-проект №1, готов).

## 1. Цель и границы

Зашифрованное локальное хранилище МЫС Desktop: зашифрованная целиком SQLite
(SQLCipher/AES-256), вывод ключа БД из мастер-пароля, репозитории для сообщений,
контактов, ключей, диалогов и состояния Double Ratchet, миграции схемы.

**Главный инвариант:** ничего секретного не попадает на диск в открытом виде.
Всё содержимое БД защищено ключом, выведенным из мастер-пароля; без пароля БД
криптографически нечитаема.

**Границы модуля (соблюдать строго, согласно CLAUDE.md):**
- Хранилище **не знает** о сети и UI.
- Хранилище **не знает** деталей крипто-протокола: оно персистит непрозрачные
  байты. Сериализацию состояния ratchet предоставляет `mys_crypto` (storage
  хранит `bytes`, не разбирает их структуру).
- Крипто-ядро **не знает** о хранилище и БД.

**Вне объёма этого под-проекта:** UI разблокировки (под-проект №3), сетевой
синк с сервером (№6), групповые диалоги, оффлайн-доставка.

## 2. Стек и зависимости

- **БД:** `sqlcipher3-binary>=0.6` — колесо с вшитым SQLCipher 4 (системная
  библиотека `sqlcipher` не требуется; проверено: в окружении её нет, а
  `sqlcipher3-binary 0.6.0` ставится готовым wheel). API совместимо с `sqlite3`
  из stdlib.
- **KDF:** `primitives.argon2id` из крипто-ядра (Argon2id).
- **Тесты:** `pytest`.

Добавляется новый пакет `src/mys_storage/` рядом с `src/mys_crypto/`. Текущая
секция `[tool.setuptools.packages.find] where = ["src"]` подхватит оба пакета.

## 3. Модель хранения: зашифрованная БД + открытый sidecar

Чтобы вывести ключ БД, нужны соль и параметры Argon2id **до** открытия БД —
значит, их нельзя держать внутри самой зашифрованной БД. Поэтому хранилище — это
**пара файлов**:

```
<vault>.db        — зашифрованная SQLCipher-БД (всё содержимое)
<vault>.meta.json — открытый sidecar: только несекретные KDF-параметры и счётчики
```

### 3.1 Sidecar (открытый, `<vault>.meta.json`)

Содержит **только несекретные** данные:

```jsonc
{
  "format_version": 1,
  "kdf": {
    "algo": "argon2id",
    "salt": "<base64, 16 байт случайной соли>",
    "time_cost": 3,
    "memory_cost": 262144,   // KiB = 256 MiB; калибруется под десктоп
    "parallelism": 4,
    "hash_len": 32
  },
  "attempts": {
    "failed": 0,
    "lockout_until": null    // unix-время; пока > now — вход заблокирован
  },
  "duress": {
    "wipe_enabled": false,   // по умолчанию выкл (см. §6)
    "threshold": 10
  }
}
```

**Почему открытый sidecar безопасен.** Соль и параметры Argon2id — не секрет (это
стандарт для хэширования паролей); их знание не даёт преимущества без пароля.
Ключ БД в sidecar **не хранится**, верификатор-секрет тоже не нужен (см. §4).
Счётчик попыток в открытом файле может быть подделан атакующим с доступом к
диску — но такой атакующий всё равно скопирует `.db` и будет брутфорсить офлайн,
поэтому лимит попыток — это **онлайн/UX-защита, а не криптогарантия**. Это честно
фиксируется в коде и доках.

### 3.2 Проверка пароля без отдельного верификатора

Правильность пароля проверяется самим SQLCipher: при неверном ключе первая же
операция чтения (`SELECT count(*) FROM sqlite_master`) бросает
`DatabaseError`/`OperationalError`. Отдельный хранимый верификатор не нужен и не
создаётся (меньше поверхности утечки).

## 4. Жизненный цикл vault (мастер-пароль)

| Операция | Шаги |
|----------|------|
| **Создание** | случайная соль (16 б) → выбор параметров → `argon2id(password, salt) → key (32 б, в SecureBytes)` → `PRAGMA key` → накат начальной миграции → запись sidecar. |
| **Разблокировка** | чтение sidecar → проверка `lockout_until` → `argon2id(password, salt) → key` → `PRAGMA key` → `SELECT count(*) FROM sqlite_master`. Успех ⇒ пароль верный, сброс `failed`. Ошибка ⇒ неверный пароль (см. §5). |
| **Смена пароля** | при открытой БД: новая соль → новый ключ → `PRAGMA rekey` → перезапись sidecar. |
| **Закрытие** | `wipe()` ключа БД и производных ключей в памяти (см. §7), закрытие соединения. |

Ключ БД на всём пути живёт в `SecureBytes` (§7) и затирается после передачи в
`PRAGMA key`/`rekey` и при закрытии vault.

## 5. Ограничение попыток (прогрессивная задержка)

По умолчанию (v1): **прогрессивная задержка / временная блокировка**, без
уничтожения данных.

- При неверном пароле: `failed += 1`; `lockout_until = now + delay(failed)`, где
  `delay` растёт (напр. экспоненциально с потолком, скажем `min(2^(failed-k), CAP)`
  секунд). Значения persist в sidecar — задержка переживает перезапуск.
- При попытке входа во время блокировки — `VaultLocked` с временем до разблокировки.
- При успешном входе — `failed = 0`, `lockout_until = null`.

## 6. Duress-wipe (опция, выключена по умолчанию)

Закладывается как **настраиваемый флаг** `duress.wipe_enabled` в sidecar и
реализуется **отдельным шагом** плана:

- Если `wipe_enabled` и `failed >= threshold` → необратимое затирание: перезапись
  и удаление `.db` и sidecar (включая соль) → БД навсегда нечитаема.
- По умолчанию выключено: риск потери данных при забытом пароле (recovery в
  анонимном мессенджере намеренно отсутствует). Включение — осознанный выбор
  пользователя (UI — под-проект №3).

## 7. Зануление ключевого материала (`SecureBytes`) — follow-up крипто-ядра

Адресует follow-up «зануление ключей» из ревью крипто-ядра.

- Новый тип `mys_crypto.secure.SecureBytes` поверх `bytearray`: контекст-менеджер,
  `.wipe()` (перезапись нулями), доступ к сырым байтам только для передачи в
  примитивы. Размещается в крипто-ядре как нижнем слое, чтобы им мог пользоваться
  и ratchet, и хранилище.
- Применение в этом под-проекте:
  - ключ БД и промежуточный вывод Argon2id — целиком через `SecureBytes`, затирание
    после использования и при закрытии vault;
  - `RatchetState.wipe()` — best-effort затирание ключевых полей при выгрузке/закрытии
    сессии.
- **Честное ограничение:** `bytes` в Python иммутабельны и могут копироваться GC;
  гарантия — best-effort. Полный перевод всех цепочечных ключей ratchet на
  `SecureBytes` — инкрементальный путь, заявляется, но не обязателен для v1; это
  фиксируется в доках.

## 8. Схема БД (SQLCipher)

Все таблицы — внутри зашифрованной БД. Уточнение к разделу 6 дизайн-документа:
**параметры Argon2id живут в sidecar, а не в `settings`** (их надо знать до
открытия БД).

- **`settings`** — `key TEXT PRIMARY KEY, value BLOB`. Хранит `schema_version`,
  выбранный режим, прикладные настройки. (KDF-параметры здесь НЕ хранятся.)
- **`identities`** — `id, kind` (ed25519/x25519, longterm/ephemeral),
  `public_key BLOB, private_key BLOB, label, context, created_at`. Приватные
  ключи защищены шифрованием всей БД.
- **`contacts`** — `id, public_key BLOB, fingerprint TEXT, alias TEXT,
  created_at, last_seen`. Сохранённые пиры для повторных сессий.
- **`conversations`** — `id, mode` (decentralized/centralized),
  `peer_contact_id` (nullable FK), `room_id BLOB` (P2P/централизованная комната),
  `title, created_at, archived`.
- **`messages`** — `id, conversation_id FK, direction` (in/out), `body BLOB`
  (расшифрованный текст — защита обеспечивается шифрованием всей БД), `status`
  (sending/sent/delivered/read/failed), `wire_seq, sent_at, received_at`.
  Производные пер-сообщенные ключи **не хранятся**.
- **`ratchet_state`** — `conversation_id PK/FK, state_blob BLOB, updated_at`. Одна
  строка на диалог; `state_blob = ratchet.serialize_state(state)` —
  непрозрачные байты для хранилища.

### 8.1 Атомарность приёма сообщения

Крипто-ядро уже коммитит состояние ratchet только после успешной расшифровки
(коммит `a35b5b4`). Хранилище обязано сохранить это согласованно: приём входящего
сообщения = **одна транзакция БД** для (1) вставки сообщения и (2) обновления
`ratchet_state`. Иначе возможен рассинхрон «сообщение сохранено, состояние нет»
(или наоборот). Экспонируется как `vault.transaction()` / транзакционный
помощник приёма.

## 9. Сериализация состояния ratchet + глобальный кап `mkskipped`

Адресует follow-up «лимит на `mkskipped`» и потребность хранилища персистить
состояние.

- В `mys_crypto.ratchet` добавляются `serialize_state(state) -> bytes` и
  `deserialize_state(blob) -> RatchetState` (включая `mkskipped`, `dhs/dhr`, `rk`,
  `cks/ckr`, счётчики `ns/nr/pn`). Инвариант: `deserialize(serialize(s))`
  продолжает корректно шифровать/расшифровывать (диалог переживает перезапуск).
- **Глобальный кап:** константа `MAX_SKIP_SESSION` — потолок суммарного числа
  пропущенных ключей за сессию. При превышении — вытеснение самых старых ключей
  (FIFO; `dict` сохраняет порядок вставки в Python 3.7+). Существующий
  пошаговый `MAX_SKIP` остаётся. Защищает память и хранилище от раздувания
  `state_blob` при злонамеренно больших `n`.

## 10. Публичный интерфейс (эскиз)

```
mys_storage.create_vault(path, password, *, params=None) -> Vault   # raises VaultExists
mys_storage.open_vault(path, password) -> Vault                     # raises WrongPassword, VaultLocked
Vault.change_password(old, new) -> None
Vault.transaction() -> ctx                                          # атомарный блок
Vault.close() -> None                                              # wipe ключей

# репозитории на Vault:
Vault.settings      # get/set ключ-значение, schema_version
Vault.identities    # CRUD ключей
Vault.contacts      # CRUD пиров
Vault.conversations # CRUD диалогов
Vault.messages      # add/list по диалогу, обновление статуса
Vault.ratchet       # load_state(conv_id)->RatchetState|None ; save_state(conv_id, state)
```

Ошибки: `WrongPassword`, `VaultLocked`, `VaultExists`, `MigrationError`,
`CorruptVault` в `mys_storage.errors`.

## 11. Миграции схемы

- `schema_version` хранится в `settings`.
- Упорядоченный список миграций `(version, up_sql)`; раннер применяет все с
  `version > current` в одной транзакции, поднимая `schema_version`.
- Начальная миграция v1 создаёт все таблицы из §8.
- Тест: повторный `open_vault` существующей БД не пере-применяет миграции; БД
  старой версии корректно мигрирует.

## 13. Тестирование

- **Нечитаемость:** созданная БД не открывается без пароля / с неверным паролем
  (raw-чтение файла не содержит открытого текста сообщений).
- **Жизненный цикл:** create → close → open (верный/неверный пароль); rekey;
  смена пароля с последующим открытием новым.
- **Лимит попыток:** неверный пароль инкрементит `failed` и ставит `lockout_until`;
  во время блокировки — `VaultLocked`; успех сбрасывает счётчик. Duress-wipe (при
  включённом флаге) затирает БД и sidecar.
- **Репозитории:** CRUD для settings/identities/contacts/conversations/messages.
- **Ratchet-персист:** `deserialize(serialize(state))` сохраняет работоспособность;
  полный диалог переживает «перезапуск» (состояние каждый шаг грузится из БД);
  атомарность приёма (сбой в транзакции не оставляет рассинхрона).
- **Глобальный кап `mkskipped`:** превышение `MAX_SKIP_SESSION` вытесняет старейшие
  ключи; `state_blob` не растёт неограниченно.
- **`SecureBytes`:** `.wipe()` зануляет буфер; контекст-менеджер затирает на выходе.
- **Миграции:** применяются по порядку, идемпотентны, старая версия мигрирует.

Разработка крипто-дополнений (§7, §9) и хранилища — через **TDD** (тест до
реализации), согласно правилам проекта.

## 14. Открытые вопросы / отложено

- Полный перевод всех цепочечных ключей ratchet на `SecureBytes` (сейчас
  best-effort) — инкрементально.
- Конкретная калибровка параметров Argon2id под целевое железо — на этапе
  реализации/профилирования.
- Точные поля для централизованного синка (токены, курсоры) — уточняются в
  под-проекте №6; в v1 схема оставляет место (`settings`, `identities.context`).
