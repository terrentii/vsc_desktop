# МЫС Desktop — дизайн-документ

**Дата:** 2026-05-29
**Статус:** утверждён (фаза дизайна)

## 1. Что это

МЫС Desktop — настольный анонимный мессенджер с двумя режимами работы.
Главный упор — **анонимность и безопасность**.

- **Централизованный режим** — клиент к существующему серверу `soufos.ru`
  (репозиторий веб-версии: https://github.com/terrentii/vsc_web). Пользователь
  входит в аккаунт МЫС, содержимое синхронизируется с веб-версией: сообщения,
  переключение между комнатами/группами — всё как на сайте, но через приложение.
- **Децентрализованный режим** — анонимный P2P-чат 1:1 без аккаунтов. Два
  пользователя вводят общую секретную фразу и получают защищённый канал. Данные
  хранятся только на компьютерах пользователей в зашифрованном виде.

## 2. Стек

- **Язык:** Python 3.13 (совпадает с веб-версией на Flask, venv уже настроен).
- **GUI:** PySide6 (Qt).
- **Сеть:** `asyncio` сокеты для P2P-транспорта; `httpx`/`websockets` для
  централизованного режима.
- **Крипто:** `cryptography`, `argon2-cffi`, реализация PAKE.
- **Хранилище:** `sqlcipher3` (зашифрованная SQLite).

## 3. Архитектура

Тонкий UI поверх набора изолированных модулей. Режимы переключаются «на лету»,
но используют общие модули крипто и хранилища.

```
┌─────────────────────────────────────────────┐
│  UI-слой (PySide6): окна, чаты, вход,         │
│  ввод секретной фразы, разблокировка БД        │
└───────────────┬───────────────────────────────┘
                │
┌───────────────▼───────────────────────────────┐
│  Контроллер приложения (оркестрация режимов,    │
│  сессии, состояние)                             │
└───┬───────────────────────────┬────────────────┘
    │                           │
┌───▼───────────────┐   ┌───────▼─────────────────┐
│ Централизованный   │   │ Децентрализованный        │
│ модуль:            │   │ модуль:                   │
│ • REST-клиент      │   │ • Rendezvous-клиент       │
│ • WebSocket-клиент │   │ • NAT traversal + relay   │
│ • токен-авторизация│   │ • PAKE-хендшейк           │
│ • синк с сервером  │   │ • транспорт (шифр. канал) │
└───┬────────────────┘   └───────┬───────────────────┘
    │                            │
    └──────────┬─────────────────┘
               │
    ┌──────────▼──────────┐   ┌──────────────────────┐
    │ Крипто-модуль        │   │ Хранилище (SQLCipher) │
    │ (примитивы + Double  │   │ репозитории: сообщения,│
    │ Ratchet + мат.слой)  │   │ контакты, ключи, настр.│
    └──────────────────────┘   └──────────────────────┘
```

**Принцип изоляции.** Каждый модуль имеет одну зону ответственности и общается
через чёткий интерфейс. Крипто и хранилище ничего не знают о сети и UI; сетевые
модули не знают деталей шифрования — они получают готовый «защищённый канал».

**Серверная часть** разрабатывается в репозитории `vsc_web` (отдельно от
десктопа, но проектируется согласованно). К серверу добавляются: REST API +
WebSocket + rendezvous-эндпоинт + relay. Сервер можно дорабатывать свободно
(вплоть до полной замены), приоритет — качественный продукт.

## 4. Крипто-протокол «МЫС-Crypto»

| Слой | Примитив | Зачем |
|------|----------|-------|
| Пароли/мастер-ключ | Argon2id | вывод ключа из мастер-пароля и из секретной фразы |
| Обмен ключами | X25519 (ECDH) | согласование общего секрета |
| Аутентификация фразой | PAKE (CPace/SPAKE2) | из короткой фразы → стойкий ключ + защита от MITM |
| Подписи | Ed25519 | подпись эфемерных ключей внутри сессии |
| Симметричное шифрование | ChaCha20-Poly1305 (AEAD) | шифрование + аутентификация каждого сообщения |
| Расщепление ключей | HKDF-SHA256 | вывод подключей из общего секрета |
| Forward secrecy | Double Ratchet | новый ключ на каждое сообщение, прошлое не расшифровать |
| Мат. слой | МЫС-transform | внешняя обёртка-перестановка (см. ниже) |

### Почему так, а не самописный шифр

Самописные шифры почти всегда взламываются — это причина №1 провала защищённых
мессенджеров. Поэтому **гарантию безопасности дают проверенные примитивы**, а
собственная математика добавляется как отдельный слой, который не может ослабить
основную защиту.

### МЫС-transform — собственная математика, встроенная безопасно

МЫС-transform — это **обратимая, ключевая биекция** над байтами шифротекста
(например, ключевая диффузия/перестановка на модульной арифметике + ключевой
S-box; ключ выводится отдельным `HKDF`-`info`, независимым от ключа AEAD).

**Инвариант безопасности:** слой применяется **ПОСЛЕ** AEAD-шифрования на
отправителе и снимается **ДО** AEAD-проверки на получателе. Поскольку это
биекция, худший случай (атакующий полностью обратил мат. слой) сводится к
обычному ChaCha20-Poly1305 — слой физически не может ослабить защиту, только
добавляет уникальности и работы атакующему.

**Требования к МЫС-transform (обязательные):**
- биективность (не теряет информацию, не создаёт коллизий);
- ключевой режим (ключ из независимого HKDF-`info`);
- применяется поверх готового AEAD-шифротекста, не вместо него;
- покрыт юнит-тестом на обратимость (decode(encode(x)) == x для всех входов).

## 5. Потоки данных

### Запуск приложения
1. Ввод мастер-пароля → Argon2id → ключ → открытие SQLCipher-БД. Без пароля БД
   нечитаема. Ограничение числа попыток.

### Децентрализованная сессия (1:1, только онлайн)
1. Оба пользователя вводят общую секретную фразу.
2. `room_id = HKDF(BLAKE2(phrase))` → регистрация на rendezvous. Сервер не знает
   фразу, видит только непрозрачный `room_id`.
3. Сервер сводит двух пиров в одной «комнате», обменивает кандидатами
   подключения (IP/порт).
4. NAT hole punching; при провале — relay через сервер (трафик уже зашифрован
   end-to-end, сервер не читает содержимое).
5. PAKE по фразе поверх транспорта → взаимная аутентификация + сессионный ключ.
   Защита от MITM: сервер фразу не видит, подменить ключи незаметно нельзя.
6. Переход на Double Ratchet; каждое сообщение → AEAD + МЫС-transform.
7. Сообщения пишутся в локальную SQLCipher-БД.

### Централизованная сессия
1. Логин (имя/пароль) → сервер выдаёт токен.
2. REST: список комнат, история. WebSocket: подписка, real-time приём/отправка.
3. Кэш/синк в локальную БД (история доступна офлайн для чтения).

## 6. Схема локальной БД (SQLCipher)

- `settings` — версия схемы, параметры Argon2id, выбранный режим.
- `identities` — локальные эфемерные/долгосрочные ключи (для централизованного
  аккаунта).
- `contacts` — сохранённые пиры (публичный ключ, отпечаток, имя-алиас) для
  повторных сессий.
- `conversations` — диалоги (режим, peer/room).
- `messages` — id, conversation_id, направление, тело, статус, timestamp.
- `ratchet_state` — состояние Double Ratchet по каждому диалогу.

Вся БД шифруется целиком (AES-256), метаданные истории защищены, ничего не
пишется на диск в открытом виде.

## 7. Обработка ошибок

- Неверный мастер-пароль — без утечки информации, ограничение попыток.
- Провал PAKE = неверная фраза или MITM → разрыв соединения с явным
  предупреждением пользователю.
- Недоступность пира → таймаут и понятное сообщение.
- Провал hole punching → автоматический fallback на relay.
- Обрыв сети → переподключение с сохранением состояния Double Ratchet.

## 8. Тестирование

- **Юнит-тесты крипто:** тест-векторы примитивов, обратимость МЫС-transform,
  корректность Double Ratchet. Разработка крипто — через TDD.
- **Интеграционные:** два локальных клиента через тестовый rendezvous (проверка
  путей hole-punch и relay).
- **Тесты хранилища:** миграции схемы, проверка что БД нечитаема без ключа.
- **Негативные:** неверная фраза, MITM-подмена ключей, битые/повторённые пакеты.

## 9. Объём v1 и что отложено

**В v1:**
- Оба режима (централизованный и децентрализованный).
- Децентрализованный режим: только 1:1, только онлайн.
- Локальное шифрованное хранилище, мастер-пароль.
- Полный крипто-протокол с МЫС-transform.

**Отложено на следующие этапы:**
- Групповые чаты в децентрализованном режиме (sender keys / MLS).
- Оффлайн-доставка в децентрализованном режиме (store-and-forward через relay).
- Передача файлов/медиа в P2P.

## 10. Декомпозиция на под-проекты (порядок реализации)

Каждый под-проект получит отдельный цикл спецификация → план → реализация:

1. **Крипто-ядро** — примитивы, МЫС-transform, Double Ratchet (с TDD и
   тест-векторами). Фундамент, ни от чего не зависит.
2. **Локальное хранилище** — SQLCipher, схема, репозитории, мастер-пароль.
3. **UI-каркас** — окна PySide6, разблокировка БД, переключение режимов.
4. **Децентрализованный модуль** — rendezvous-клиент, NAT traversal + relay,
   PAKE, транспорт (зависит от крипто-ядра).
5. **Серверные доработки** — rendezvous + relay в `vsc_web`.
6. **Централизованный модуль** — REST + WebSocket клиент, синк (зависит от
   серверных доработок).
