# CLAUDE.md — МЫС Desktop

Руководство для работы над проектом. Полный дизайн:
`docs/superpowers/specs/2026-05-29-mys-desktop-design.md`.

## Что это за проект

МЫС Desktop — настольный анонимный мессенджер (Python + PySide6) с двумя
режимами. Главный приоритет — **анонимность и безопасность**, всё остальное им
подчинено.

- **Централизованный режим** — клиент к серверу `soufos.ru` (веб-версия:
  https://github.com/terrentii/vsc_web, Flask + SQLAlchemy + SQLite). Вход в
  аккаунт, синхронизация комнат и сообщений с веб-версией.
- **Децентрализованный режим** — анонимный P2P-чат 1:1 без аккаунтов: общая
  секретная фраза → защищённый канал. Данные только на устройствах пользователей,
  в зашифрованном виде.

## Архитектура (модули)

Тонкий UI поверх изолированных модулей. Режимы используют общие крипто и
хранилище.

- **UI-слой (PySide6)** — окна, чаты, вход, ввод фразы, разблокировка БД.
- **Контроллер приложения** — оркестрация режимов, сессии, состояние.
- **Централизованный модуль** — REST-клиент, WebSocket-клиент, токен-авторизация,
  синк с сервером.
- **Децентрализованный модуль** — rendezvous-клиент, NAT traversal + relay,
  PAKE-хендшейк, шифрованный транспорт.
- **Крипто-модуль** — примитивы + Double Ratchet + МЫС-transform.
- **Хранилище (SQLCipher)** — репозитории: сообщения, контакты, ключи, настройки.

**Серверная часть** (rendezvous + relay + REST + WebSocket) ведётся в репозитории
`vsc_web`, но проектируется согласованно. Сервер можно дорабатывать свободно.

### Границы модулей (соблюдать строго)

- Крипто и хранилище **не знают** о сети и UI.
- Сетевые модули **не знают** деталей шифрования — получают готовый защищённый
  канал.
- Каждый модуль — одна зона ответственности, общение через явный интерфейс.

## Крипто-протокол «МЫС-Crypto»

Гарантию безопасности дают **проверенные примитивы**; собственная математика —
отдельный слой, который не может ослабить защиту.

- **Argon2id** — мастер-пароль и секретная фраза.
- **X25519** — обмен ключами. **Ed25519** — подписи.
- **PAKE (CPace/SPAKE2)** — из фразы стойкий ключ + защита от MITM.
- **ChaCha20-Poly1305** — AEAD-шифрование сообщений.
- **HKDF-SHA256** — расщепление ключей.
- **Double Ratchet** — forward secrecy (новый ключ на сообщение).
- **МЫС-transform** — собственный мат. слой.

### МЫС-transform — правила (НАРУШАТЬ НЕЛЬЗЯ)

Обратимая ключевая биекция над байтами шифротекста. Применяется **ПОСЛЕ** AEAD на
отправителе, снимается **ДО** AEAD-проверки на получателе. Так худший случай
сводится к обычному ChaCha20-Poly1305 — слой не может ослабить защиту.

- биективность (без потерь и коллизий);
- ключ из независимого HKDF-`info`;
- поверх AEAD-шифротекста, не вместо него;
- юнит-тест: `decode(encode(x)) == x`.

## Правила безопасности проекта

- **Не изобретать собственные шифры** как единственную защиту. Только проверенные
  примитивы для гарантий; своя математика — поверх, не вместо.
- БД всегда зашифрована (SQLCipher/AES-256). Ничего секретного на диск в открытом
  виде.
- Сервер в децентрализованном режиме **не должен** видеть секретную фразу или
  открытый текст сообщений (только непрозрачный `room_id` и E2E-шифротекст).
- Провал PAKE = неверная фраза или MITM → разрыв с предупреждением.
- Ограничение попыток ввода мастер-пароля.

## Объём v1

В v1: оба режима; децентрализованный — только 1:1 и только онлайн; локальное
шифрованное хранилище; полный крипто-протокол.

Отложено: групповые P2P-чаты, оффлайн-доставка (store-and-forward), передача
файлов в P2P.

Каждый под-проект — отдельный цикл спецификация → план → реализация:

2. **Локальное хранилище** (SQLCipher). ✅ Реализовано в `src/mys_storage/`
   (зашифрованный vault на `sqlcipher3-binary`, открытый sidecar с KDF-параметрами,
   мастер-пароль через Argon2id, прогрессивная задержка + опц. duress-wipe, миграции,
   репозитории, атомарный приём сообщения). Спека/план:
   `docs/superpowers/specs/2026-05-29-local-storage.md`,
   `docs/superpowers/plans/2026-05-29-local-storage.md`.
3. **UI-каркас** (PySide6). ✅ Реализовано в `src/mys_ui/` (контроллер без Qt поверх
   хранилища, окно входа создание/разблокировка, главное окно: верхняя панель
   `Центр|P2P` + две панели список/чат, минимальная тёмная тема Fusion). Сеть/PAKE/
   ratchet-транспорт — заглушки до под-проекта №4; тесты headless через
   `QT_QPA_PLATFORM=offscreen`. Спека: `docs/superpowers/specs/2026-05-29-ui-shell.md`.
4. **Децентрализованный модуль** (зависит от крипто-ядра). ✅ Реализовано в
   `src/mys_decentralized/` и `src/mys_crypto/pake.py`: CPace на ristretto255
   (libsodium через ctypes, `_ristretto.py`), деривация `room_id`/`prs`, wire-протокол
   и фрейминг, хендшейк (CPace+key-confirmation → `sk` + init ratchet по роли),

## Разработка

- Виртуальное окружение: `.venv/` (Python 3.13).
- Крипто-код разрабатывается через **TDD** (тесты до реализации).

## Известные follow-up по крипто-ядру (для будущих под-проектов)

Не баги, а технический долг, выявленный на ревью; адресовать в смежных под-проектах:

- ✅ **Лимит на `mkskipped`.** Добавлен глобальный кап `MAX_SKIP_SESSION` с FIFO-вытеснением
  старейших ключей (под-проект №2, `ratchet.py`).
- ◻️ **Зануление ключей** (частично). Введён `SecureBytes` (`mys_crypto/secure.py`),
  применён к ключу БД в `mys_storage`; есть best-effort и `RatchetState`. Остаётся долг:
  полный перевод всех цепочечных ключей ratchet (`sk`, `rk`, `ck`, `mk`) на `SecureBytes`.
  Ограничение принципиальное: `bytes` в Python иммутабельны и копируются GC.
- **Детерминизм transform.** Transform-ключ статичен на сессию → внешний слой это
  фиксированная публичная биекция (одинаковые байты AEAD-шифротекста на одной позиции
  шифруются одинаково). AEAD это не ослабляет, но транспорт не должен полагаться на
  transform для защиты от анализа трафика.
- ✅ **Вход для ratchet.** Закрыт под-проектом №4: `mys_decentralized.handshake`
  выводит `sk` и стартовый DH-ключ Боба из `ISK` (CPace) и инициализирует ratchet
  по роли. `ratchet_init_alice/bob` остаются без изменений.
- ✅ **Порядок отправки в Double Ratchet.** Закрыто в `session.py`/`service.py`:
  INITIATOR при подключении шлёт prime-кадр (пустой payload, `Session.send_prime`),
  из которого RESPONDER выводит свою отправляющую цепочку; пустой payload на приёме
  не персистится/не показывается, лишь двигает ratchet. Исходящие у не-готового
  RESPONDER (`Session.can_send is False`) пишутся как `pending` и ставятся в очередь
  (`_outbox`), флашатся при первом входящем. Снимает дедлок и `kdf_ck(None)`.
- **Доступ к vault из потоков.** `mys_storage` открывает соединение с
  `check_same_thread=False` и сериализует доступ общим `RLock` (`_LockedConnection`),
  чтобы поток UI (чтение) и фоновый поток `P2PService` (запись) делили один Vault.

## Зависимости и тесты под-проекта №4

- **PyNaCl не используется** для ristretto255 (его колесо не экспонирует символы);
  CPace работает с libsodium напрямую через `ctypes` (`mys_crypto/_ristretto.py`).
  Требуется системный libsodium ≥1.0.18.
- **`pytest-asyncio`** (dev) + `asyncio_mode = auto` в **`pytest.ini`** (он перекрывает
  `[tool.pytest.ini_options]` в `pyproject.toml` — настройки держим в `pytest.ini`).
- Интеграционные тесты гоняют блокирующие методы сервиса через `asyncio.to_thread`,
  чтобы не застопорить event loop теста (на нём живёт rendezvous-сервер).

## Зависимости и тесты под-проекта №6

- **`httpx`** (REST-клиент) и **`websockets`** (WS-клиент) — рантайм-зависимости в
  `pyproject.toml`. Фикстура `fast_kdf` — в `tests/conftest.py` (быстрый Argon2id).
- Интеграционные тесты сервиса — фейковый сервер in-process: REST через
  `httpx.MockTransport`, WS через `websockets.serve` на loopback. **Все** блокирующие
  методы `CentralizedService` (в т.ч. `stop`) гонять через `asyncio.to_thread`: они
  делают `fut.result()` и заблокируют event loop теста, на котором живёт WS-сервер,
  — иначе дедлок при закрытии WS (close-handshake ждёт ответа замороженного сервера).
- **UI-тесты headless**: переключать режим через `controller.set_mode`, а не
  `top._select` — последний триггерит реальный `_on_mode`, который при отсутствии
  сессии открывает **модальный** диалог входа (`exec()` виснет в offscreen). Сам
  вызов формы входа на смене режима проверяется через monkeypatch `_central_login`.
- **Граница UI/контроллер.** Контроллер без Qt: владеет фабрикой и жизненным циклом
  `CentralizedService`, раздаёт его колбэки наблюдателям; UI-слой подписывается и
  маршалит их в Qt через `_CentralBridge` (signal.emit из потока сервиса → слот в
  UI-потоке). Форма входа открывается только если `controller.central_available()`
  (есть фабрика) — иначе режим переключается как локальная заглушка.

## Известные follow-up по централизованному модулю (№6)

- **Порядок отправки у RESPONDER в №4 здесь неактуален** — централизованный режим
  без ratchet, шлёт любой участник.
- ✅ **Создание комнаты из десктопа.** `POST /api/rooms {name}` (Bearer) на сервере
  (`vsc_web/centralized.py`): 10-значный `room_id`, `is_open=True`, caller —

  маршрутизирует в режиме «Центр» с активной сессией. `+ Новый диалог` теперь
  заводит реальную серверную комнату, отправка в неё работает. Управление
  (приглашения/выход/удаление) из десктопа — вне объёма v1.
- ✅ **`resume()` при старте.** `MainWindow.__init__` вызывает `_try_auto_resume`:
  если в vault есть сохранённая сессия (`controller.central_has_saved_session()` —

  результат — сигналом `_CentralBridge.resumed` в UI-поток. Повторный вход не нужен;
  переключение в «Центр» с восстановленной сессией не открывает форму входа.
- ✅ **Очистка локального кэша при логауте** — по настройке `central.wipe_on_logout`
  (по умолчанию выкл: история остаётся для офлайн-чтения). `account.wipe_local_cache`

  `service._logout` вызывает её при выставленном флаге. UI: чекбокс «Стирать историю
  «Центра» при выходе» + кнопка «Выйти из аккаунта» в настройках. Примитивы удаления
  добавлены в `mys_storage` (`SettingsRepo.delete/delete_prefix`, `ConversationsRepo.delete`,
